# SPA

Single Page Application

너무 자연스럽게 SPA형태로 개발을 하고 있지만, 누군가가 'SPA가 뭔지 설명해보세요'라고 물어본다면 뭐라고 답해야할지 모르겠다.

`어린 아이도 이해할 수 있도록 잘 설명할 수준으로 알아보자`

## 과거에 웹페이지가 렌더링 되던 형태

과거에는 지금보다 문서 하나에 전달되는 파일의 용량이 작았다.

그래서 어떤 요소를 클릭해서 페이지가 이동될 때, 완전히 새로운 페이지를 서버에서 전송해주는 형태로 렌더링하였다.

하나의 웹페이지는 여러 개의 HTML 로 구성되어 있는 형태였던 것이다.

`MainPage.html`, `LoginPage.html` 이런 느낌이 아니었을까?

이런 형태는 곧, 웹 어플리케이션의 뷰를 서버에서 담당했었다는 의미이다.(더 공부가 필요한 부분으로 보임)

하지만 현대에 가까워지면서 웹사이트는 점차 전달하는 데이터의 양이 많아졌고, 한 페이지에 담겨있는 용량이 커지면서 매번 새로운 페이지를 전달하는 것이 비효율적이고, 많은 자원을 소모하게 되었다.

## SPA란?

이런 문제를 해결하고자 등장한 개념이 SPA(Single Page Application)이다. SPA 는 하나의 모던 웹 패러다임이라고도 말할 수 있겠다.

SPA 는 웹 사이트에 필요한 정적 리소스를 처음에 한 번 다운로드 한다. 즉 최초 웹사이트에 대한 호출이 이루어졌을 때, 전체 페이지를 담고 있는 **HTML 하나**를 서버로 부터 받아서 클라이언트 사이드에서 렌더링한다.(저장이 맞나?) 그리고 이후에는 서버로부터 필요한 데이터만 JSON으로 받아서 페이지를 갱신하는 형태로 렌더링 된다. 이 때, 페이지 갱신에 필요한 재 렌더링은 서버가 아니라 웹 브라우저에서 이루어진다.

(기존의 MPA - Multiple Page Application 같은 경우 새로운 페이지 요청 시 마다 정적 리소스(HTML)을 서버로부터 다시 받고, 페이지를 다시 렌더링하는 형태로 구현된다 -> 페이지가 깜빡 한다는 의미...)

즉 최초 로딩땐 실행된 HTML은 파일은 계속 렌더링 되어 있는 형태에서, 변경되는 부분만 다시 렌더링 해주는 형태로, 전체의 변경이 필요하지 않고 필요한 부분만 효율적으로 변경이 가능한 점이 큰 장점이다.

리액트를 생각해보면, `root`라는 id를 가지는 div 태그 하나를 가지는 HTML을 기본으로 해서, 해당 div 태그 안에 렌더링 되는 페이지를 상황에 맞춰서 보여지는 형태를 말하는 듯 하다.

새로고침이 되지 않고(페이지 깜박임이 없이) 변경된다는 점에서 좋은 UX를 제공할 수 있다.

하지만 장점만이 있는 것은 아니다.

**초기 구동 속도가 느리다는 단점이 있다.**

최초 실행 시에 모든 정적 리소스를 불러온다고 했다. 이 때 이 정적 리소스의 규모가 크면 어떻게 될까? 자연스럽게 로딩 시간이 길어질 것이다. 즉 정적 리소스의 크기에 따라서 최초 로딩 시간이 길어질 수 있다는 단점이 존재한다.

**SEO에 취약하다는 단점이 있다.**

SPA 형태로 빌드된 결과물은, `body`태그가 비어있고 이를 JavaScript를 통해서 동적으로 교체하는 방법으로 동작한다. 즉 `검색엔진은 해당 페이지를 비어있는 페이지로 인식한다`(검색엔진이 크롤링을 할때 JavaScript를 실행하지 않고 긁어가기 때문이다. )

전통적인 MPA는 어떻다고 했었지? 매번 완성된 HTML을 그때 그때 서버로부터 받아오는 형태라고 했다. 그러므로 검색엔진이 크롤링 했을 때 필요한 정보가 다 있다고 인식해서 SEO에서 SPA에 비해서 강점이 있다.

이를 보완하기 위해서 리액트/뷰 와 같은 SPA 라이브러리들은 SSR 기능을 지원한다.

**화면 변하는 과정이 유저에게 노출될 수 있다.**

SPA는 데이터를 필요에 따라 요청해서 화면을 구성하는 형태라고 했다. 즉, 데이터 받아오는 속도나, 설계 등에 따라서 화면이 다 변하기 전 과정이 유저에게 노출될 수 있다. 이에 대한 처리가 없다면 당연히 UX는 좋지 않겠다. 적절한 로딩화면 처리 등 대처를 통해 UX 를 개선하는 작업이 추가로 요구된다는 의미이다.

초기 구동속도가 느리다는 단점, 화면 변하는 모습이 유저에게 노출 될 수 있다는 단점은 `lazy-loading`, `pre-rendering`을 통해서 해결할 수 있다.

`lazy-loading`은 간단히 말해서, 특정 URL에서 필요한 리소스를 해당 URL에 실제 진입할 때 받아오도록 처리하는 형태이다. 초기에 모든 리소스를 받아오지 않으니까 어느정도 느린 초기 구동속도 문제를 해결할 수 있다. `초기 로딩에 걸리는 시간을 라우팅 별로 분산해서 작업한다`라는 느낌으로 생각하면 되겠다.
리액트 기준으로는 `React.lazy`를 활용해서 특정 컴포넌트를 import 하는 형태가 되겠다.

`pre-rendering`은 페이지 이동 시 데이터를 요청하고 받아오는 과정이 필요한 경우에 데이터를 받아서 보여주어야 하는 UI에 대해서 빈 화면을 보여주는 대신 로딩 스피너를 보여주는 등의 처리를 하는 것을 말한다.

SPA에서는 데이터가 없는 화면이 먼저 동기적으로 렌더링 된 후, 비동기 요청을 통해서 서버에서 데이터를 받아 필요한 화면을 재구성하게 된다.

```javascript
// 아래 코드는 Suspense 컴포넌트 안의 Component로딩이 끝날 때 까지 로딩 화면을 보여주게 처리한다.
<Suspense fallback={<div>Loading...</div>}>
  <Component />
</Suspense>
```
